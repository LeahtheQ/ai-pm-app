## 목적

사용자가 제품을 직접 사용하면서 드러나는 사용성 문제를 발견하고, 개선 우선순위를 결정하기 위한 UT를 체계적으로 계획한다.

## 입력 변수

- `테스트 대상`: 제품 또는 특정 기능/플로우
- `테스트 목적`: 무엇을 발견하고 싶은가?
- `참가자 프로필`: 어떤 유형의 사용자를 대상으로 하는가?
- `테스트 환경`: 원격 / 대면, 모더레이티드 / 언모더레이티드
- `일정`: 테스트 기간

## 프레임워크

### 1. UT 목적 정의
- **핵심 질문**: "이 테스트로 답해야 하는 것은?"
  - 예: "사용자가 처음 가입 후 첫 번째 프로젝트를 혼자 만들 수 있는가?"
- **가설**: "우리는 사용자가 [X] 단계에서 [Y] 문제를 겪을 것이라 예상한다"
- **성공 기준**: 완료율, 오류율, 만족도 기준치 설정

### 2. 참가자 기준
| 항목 | 내용 |
|------|------|
| 인원 수 | 5~8명 (정성, 문제 발굴 목적) |
| 포함 기준 | [타겟 사용자 특성] |
| 제외 기준 | 경쟁사 직원, UX 리서처, 내부 팀원 |
| 다양성 고려 | 신규/기존 사용자 비율, 디지털 리터러시 수준 |

### 3. 태스크 시나리오 작성
좋은 시나리오의 특징:
- 현실적인 맥락 포함 (역할, 상황, 목표)
- 특정 UI 요소를 언급하지 않음
- 완료 기준이 명확함

**시나리오 형식**:
> [맥락] 당신은 [역할]입니다.  
> [상황] [상황 설명].  
> [태스크] [해야 할 것]을 해보세요.

| # | 태스크명 | 시나리오 | 예상 소요시간 | 완료 기준 |
|---|---------|---------|-------------|-----------|
| T1 | | | 3분 | |
| T2 | | | 5분 | |

### 4. 측정 지표
| 지표 | 정의 | 측정 방법 |
|------|------|-----------|
| 완료율 (Task Success Rate) | 태스크 성공한 참가자 비율 | 관찰 기록 |
| 오류율 (Error Rate) | 태스크당 평균 오류 횟수 | 관찰 기록 |
| 소요시간 (Time on Task) | 태스크 완료까지 걸린 시간 | 타이머 |
| SUS 점수 | System Usability Scale (0~100) | 설문 10문항 |
| 주관적 만족도 | 각 태스크 후 1~7점 평가 | 설문 |

### 5. 모더레이터 가이드
- **소개 스크립트**: 테스트 목적, 녹화 동의, "제품이 아닌 우리가 테스트받는 것" 강조
- **Think Aloud 안내**: "지금 보이는 것, 생각하는 것을 말해주세요"
- **개입 금지**: 막힌 상황에서 힌트 주지 않기 (최대 3분 대기)
- **관찰 포인트**: 망설임, 오클루전, 오류 클릭, 표정 변화

### 6. 결과 분석 계획
1. **레인보우 스프레드시트**: 참가자 × 문제 매트릭스 작성
2. **심각도 분류**:
   - Critical: 모든 참가자가 실패 또는 심각한 불만
   - Major: 50% 이상 실패 또는 큰 불편
   - Minor: 일부 참가자 이슈, 회피 가능
3. **개선 우선순위**: 심각도 × 빈도 × 수정 용이성
4. **다음 단계**: 즉시 수정 / 추가 리서치 / 백로그 등록

## 출력 형식

1. UT 계획서 요약 (1페이지 개요)
2. 참가자 스크리너
3. 태스크 시나리오 목록
4. 모더레이터 가이드 스크립트
5. 결과 기록 양식 (레인보우 스프레드시트 템플릿)

## 품질 체크리스트

- [ ] 각 태스크에 명확한 완료 기준이 있는가?
- [ ] 시나리오가 특정 UI 요소를 언급하지 않는가?
- [ ] 5~8명의 적절한 참가자 수를 계획했는가?
- [ ] SUS 포함 정량 지표와 정성 관찰 지표가 균형 잡혀있는가?
- [ ] 결과 분석 후 즉시 실행 가능한 개선안 도출 계획이 있는가?
