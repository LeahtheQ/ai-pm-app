## 목적

PM이 작성하고 개발팀, 디자인팀, 비즈니스 이해관계자가 공통으로 사용할 수 있는 완결된 PRD를 작성한다.

## 입력 변수

- `제품/기능명`: 명세 대상
- `배경`: 이 기능을 만들게 된 맥락
- `타겟 사용자`: 주요 이용자 페르소나
- `범위`: 이번 PRD에서 다루는 기능의 경계
- `출시 일정`: 목표 출시 시점 (선택)

## PRD 전체 템플릿

### 1. 메타데이터
| 항목 | 내용 |
|------|------|
| 문서 버전 | v1.0 |
| 작성자 | |
| 최종 수정일 | |
| 상태 | 초안 / 검토 중 / 승인 완료 |
| 연관 문서 | 디자인 시안, 기술 스펙 링크 |

### 2. 배경 및 문제 정의
- 왜 지금 이 기능을 만드는가?
- 어떤 사용자 문제 또는 비즈니스 기회가 있는가?
- 현재 솔루션의 한계는 무엇인가?

### 3. 목표 (측정 가능한 형태로)
- 비즈니스 목표: [지표] [방향] [수치] [기간]
  예: 유저 이탈률 20% 감소 (출시 후 90일 이내)
- 사용자 목표: 어떤 사용자 문제를 해결하는가?

### 4. 사용자 페르소나
| 항목 | 주요 페르소나 | 보조 페르소나 |
|------|-------------|--------------|
| 이름 (가상) | | |
| 직무/역할 | | |
| 핵심 목표 | | |
| 주요 불편 | | |

### 5. 핵심 사용자 스토리 (상위 5~7개)
| # | 유저스토리 | 우선순위 |
|---|-----------|----------|
| 1 | 나는 [역할]로서, [기능]을 원한다. 왜냐하면 [가치] 때문이다. | Must |
| ... | | |

### 6. 기능 요구사항
#### Must Have (없으면 출시 불가)
- FR-001: [요구사항 설명]
  - 인수기준: Given [전제] / When [행동] / Then [결과]
- FR-002: ...

#### Should Have (있으면 좋지만 없어도 출시 가능)
- FR-010: ...

#### Could Have (향후 검토)
- FR-020: ...

### 7. 비기능 요구사항
| 항목 | 요구사항 | 측정 방법 |
|------|----------|-----------|
| 성능 | 페이지 로드 3초 이내 | Lighthouse 점수 |
| 보안 | OWASP Top 10 준수 | 보안 감사 |
| 접근성 | WCAG 2.1 AA | 자동화 테스트 |
| 지원 브라우저 | Chrome, Safari, Edge 최신 2버전 | |

### 8. 성공 지표

| [지표명] | | | 출시 90일 |

## 출력 형식

위 템플릿을 채운 완성된 PRD를 마크다운으로 제공한다. 입력된 정보가 부족한 섹션은 "[PM이 채워야 할 사항]"으로 표시한다.

## 품질 체크리스트

- [ ] 목표가 측정 가능한 지표로 표현되었는가?
- [ ] 기능 요구사항에 인수기준이 포함되었는가?
- [ ] Must/Should/Could 분류가 명확한가?
- [ ] 성공 지표가 사용자 행동 지표를 포함하는가?
- [ ] 미결 사항이 명확한 담당자와 기한을 포함하는가?
