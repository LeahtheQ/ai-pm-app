## 목적

개발팀이 즉시 실행할 수 있는 명확하고 완결된 유저스토리와 인수기준을 작성한다.

## 입력 변수

- `기능/요구사항`: 유저스토리로 만들 기능 설명
- `사용자 역할`: 주요 페르소나 또는 역할
- `핵심 가치`: 이 기능으로 사용자가 얻는 것
- `기술 제약`: 알고 있는 기술적 제한 (선택)

## 프레임워크

### 1. INVEST 원칙 체크

| 원칙 | 의미 | 나쁜 예 | 좋은 예 |
|------|------|---------|---------|
| Independent | 다른 스토리와 독립적 | "로그인 후 대시보드 표시" | 로그인 / 대시보드 스토리 분리 |
| Negotiable | 구현 방식은 협상 가능 | "버튼을 파란색으로" | "사용자가 동작을 확인할 수 있다" |
| Valuable | 사용자 가치 명시 | "DB 스키마 변경" | "사용자가 저장된 설정을 볼 수 있다" |
| Estimable | 크기 추정 가능 | "AI 기능 추가" | "텍스트 요약 기능 (5~8 SP)" |
| Small | 1 스프린트 내 완료 | "전체 인증 시스템" | "이메일로 로그인" |
| Testable | 인수기준 존재 | "더 빠르게 만들기" | "로딩 시간 3초 이내" |

### 2. 유저스토리 형식

**기본 형식**:
> 나는 **[역할]**로서,  
> **[기능/행동]**을 원한다.  
> 왜냐하면 **[가치/목적]** 때문이다.

**영문 형식** (글로벌 팀):
> As a **[role]**,  
> I want **[feature/action]**,  
> So that **[value/outcome]**.

### 3. 인수기준 (Gherkin 형식)

```gherkin
Feature: [기능명]
  As a [역할]
  I want [기능]
  So that [가치]

  Scenario: [시나리오명] — 정상 케이스
    Given [초기 상태 / 전제조건]
    When [사용자 행동]
    Then [기대 결과]

  Scenario: [시나리오명] — 예외 케이스
    Given [다른 초기 상태]
    When [다른 행동]
    Then [다른 결과]
```

### 4. 스토리 포인트 가이드 (피보나치)
| 포인트 | 복잡도 | 예시 |
|--------|--------|------|
| 1 | 매우 단순 | 버튼 텍스트 변경 |
| 2 | 단순 | 기존 페이지에 필드 추가 |
| 3 | 보통 | 새 CRUD 화면 |
| 5 | 복잡 | 외부 API 연동 |
| 8 | 매우 복잡 | 새 모듈 설계 및 구현 |
| 13+ | 분할 필요 | → 작은 스토리로 분할 |

### 5. Definition of Done (완료 기준)
모든 유저스토리에 적용되는 공통 기준:
- [ ] 인수기준을 모두 통과한 테스트 코드 작성
- [ ] 코드 리뷰 완료
- [ ] QA 환경에서 테스트 통과
- [ ] 필요한 경우 문서 업데이트
- [ ] 프로덕션 배포 준비 완료

## 출력 형식

1. 유저스토리 목록 (테이블 형식, 우선순위 포함)
2. 각 스토리의 Gherkin 인수기준
3. 스토리 포인트 추정 근거
4. 분할 권고사항 (13+ 포인트 스토리)

## 품질 체크리스트

- [ ] INVEST 원칙 6개를 모두 충족하는가?
- [ ] 인수기준이 Given/When/Then 구조를 따르는가?
- [ ] 정상 케이스와 예외 케이스가 모두 포함되었는가?
- [ ] 기술 구현 방법이 아닌 사용자 가치가 중심인가?
- [ ] 한 스프린트 내 완료 가능한 크기인가?
